# Value Lab v0.6.0 页面逻辑审查

## 审查原则

所有页面遵循同一条依赖链：

```text
先确定对象
→ 再确定配置或条件
→ 再展示事实数据
→ 再计算结果
→ 最后给出比较和建议
```

任何依赖前置选择的信息，不应在其上下文之前出现。

## 一、成本页

### 不合理

1. 新品价格在机型与配置选择之前展示，用户先看到价格、后知道价格属于谁。
2. 机型、配置、买入方式、卖出方式和成本参数混在一个工具栏，混淆“对象选择”和“计算条件”。
3. 新品价格、持有成本结果、趋势和建议没有明确步骤编号。
4. 恢复默认会直接恢复到某款默认产品，但页面没有解释默认对象。

### 合理但缺少

1. 应先显示“产品与配置”。
2. 新品基准应只依赖机型与配置。
3. 持有成本条件应独立成下一步。
4. 结果、趋势、路径和结论应逐级排列。
5. 页面应明确当前计算对象。

## 二、市场页

### 不合理

1. 进入页面后直接汇总整个品类价格，缺少明确机型和配置上下文。
2. 搜索、价格类型、可信度、平台混为一组，但没有“先选择产品”的入口。
3. KPI 可能来自多个型号与配置，数字缺乏可比性。
4. 缺少数据日期筛选。

### 合理但缺少

1. 先选产品类别、机型、配置。
2. 再选价格类型、平台、可信度、日期。
3. 再展示当前摘要和记录。
4. 应在结果标题中重复当前对象，避免用户滚动后失去上下文。

## 三、购买参考页

### 不合理

1. 页面会自动选择筛选结果第一项，并立即展示详情；用户没有明确选择。
2. “产品列表”和“详细信息”之间缺少明显的依赖提示。
3. 从其他页面跳转时可以自动带入产品，但普通进入时也自动带入，二者未区分。

### 合理但缺少

1. 类别与筛选是第一步。
2. 产品列表是第二步。
3. 用户点击具体产品后，第三步才展示详情。
4. 从成本页或全局搜索带入时，可以直接展示，但应高亮来源对象。

## 四、生活平替页

### 不合理

1. 列表生成后自动选择第一条，详情无用户操作即出现。
2. 用户可能误以为详情是系统默认推荐，而非列表第一项。

### 合理但缺少

1. 搜索与筛选在前。
2. 列表在后。
3. 只有点击条目后才出现详情。
4. 过滤后若原选中项不再可见，应清空详情。

## 五、我的页面

### 不合理

1. 没有任何物品时仍显示 0 元资产 KPI，信息价值低。
2. 添加物品只保存机型，容量、内存、SSD、网络等依赖备注文本解析，估值可能错误。
3. 添加弹窗没有“产品类型 → 机型 → 配置 → 购买信息”的完整层级。
4. 汇总结果在物品记录之前，但没有明确说明它依赖已添加物品。

### 合理但缺少

1. 空数据时先显示添加入口。
2. 添加流程应按产品、机型、配置、购买信息逐级展开。
3. 保存具体配置字段，而不是从备注猜测。
4. 有物品后再显示资产汇总。

## 六、全局框架

### 已有合理设计

- 五个一级模块边界基本清楚。
- 全局搜索遵循“输入 → 结果 → 进入页面”。
- 主题、筛选和本地数据可以持久化。
- 移动端有独立底部导航。

### 仍缺少

- 所有页面统一步骤标题和编号。
- 未满足前置条件时的空状态文案。
- 跨页面带入条件时的上下文提示。
- 页面结果区重复当前选择，避免滚动失去上下文。
- 添加物品的显式配置数据。
